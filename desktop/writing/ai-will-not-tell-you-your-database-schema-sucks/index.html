AI Will Not Tell You Your Database Schema Sucks

Design the schema separately and read it yourself. The model will never tell you that your database schema sucks.

The model puts the same amount of effort into the schema as everything else. Design it alongside the rest of the system and the knock-on effects never get enough attention. Then once you're in the coding loop, the model treats the schema as fixed — which was a hundred percent true in the old way of programming. Things are different now. Everything can be changed.

Part of it is politeness. The model adapts to what it perceives as your level — ask like an amateur and the answers come back pitched at one. It will never call you stupid.

Notice where these problems come from. They're all depth-based, knock-on effects. The model can name them if you take the time to ask — but it will never bring them up during the coding phase. In the old days, a schema was the result of twenty meetings and multiple design reviews. It was sandstone. So when a bug surfaces, you get the patch, not the diagnosis:

You'll build it out, keep improving everything else, and just amass a slow pile of zero-row filters and exploding tables, until half your application code exists to compensate for the schema.

The Depth Problem

A schema is a depth-first problem. Swarms and parallel agents buy you breadth — many takes at once — but nobody in there is pushing a single chain all the way down. That's still your job.

Normally you'd work through the depth by building — that's how frontend work goes. Render it, click the buttons, fix what's wrong, repeat. You and the model do the depth together. But the schema is the one part of the system the model tries to never change, so the artifact that most needs the working-through never gets it.

It's the worst place to lose the loop, too. AI made code about 10x cheaper, so code is disposable — but data has mass. Greenfield or mid-rewrite, the schema is as cheap to change as everything else. Exactly when the model assumes it isn't.

The Solution

Design the schema as its own artifact — before any system code, but after the architecture. Paste the whole specification into ChatGPT Pro and ask what a professional would build greenfield. Even if you're no database expert, the answer comes back in the jargon of one — and since the model adapts to the level it sees, expert language in the context makes every response after it smarter.

Then read the schema yourself, column by column. Do this ten more times. Every ten-minute Pro response I've gotten still had an error in it — a knock-on effect it didn't push through, or a table that doesn't exist. From the same project:

That's what the extra rounds are for. What Pro really sells is rolls of the dice: many thinking patterns in parallel, hoping one roll lands on a schema that survives the knock-on effects. It's not thinking the chain through — it's Monte Carlo. Make the rounds adversarial — you don't have to write the attacks yourself:

You're the depth-first search.
